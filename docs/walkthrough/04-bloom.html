Chapter 04 · the signature idea

The Bloom, exactly

The Bloom is a radial flower: one petal per area, each filling as you tend it and easing back when it goes quiet. This chapter is the precise algorithm — computeAreaActivity() in components/Bloom/activity.ts — because "how is the Bloom calculated?" is the question about this app.

What one petal represents

A petal's fill is a number in [0, 1] for one area, computed over a rolling window (14 days for the live Bloom). It answers: "over the last two weeks, how fully has this area been tended, adjusted for how much it matters and dragged down by any tugs?" Two tiers combine.

Tier 1 — weighted adherence (the lift)

Each tend habit in the area contributes its own adherence = completed ÷ expected, clamped to 1, weighted by its share of the area.

for each tend habit h:
  expected = expectedCompletionsInWindow(h.cadence, 14)
  completed = distinct days h was logged in the window
  adherence = min(1, completed / expected)
positive = Σ ( (h.weight / totalWeight) × adherence )
  • Per-habit clamping means a daily habit and a "3× a week" habit are each measured against their own cadence. Doing your 3×-a-week habit 3 times = full credit, even though a daily habit would need 14.
  • Weights (habit.weight, default 1 = equal) let you say "meditation counts more toward Mind than my reading habit." The split always normalizes to 1.

Tier 2 — the tug penalty (the drag)

Every logged tug in the window subtracts, honestly but never punitively.

penalty = Σ (tug.tugWeight ?? 1)  over logged tugs in window
penaltyFrac = min(TUG_CAP 0.6, penalty × TUG_UNIT 0.12)
score = max(0, positive − penaltyFrac)

One "some" tug (weight 1) costs 0.12 of the petal; the cap of 0.6 means tugs can never erase more than 60% — you are never sent to zero by slips alone. The floor of 0 means a petal never goes negative.

Tier 3 — the importance curve (the attention dial)

Finally the score is bent by the area's importance via a gamma exponent:

fill = min(1, score γ)  where γ = { core 1.35, matters 1.0, optional 0.6 }
ImportanceγEffect
core ("really matters")1.35Fills less readily — needs more to look full.
matters1.0Linear — score is the fill.
optional ("nice to have")0.6Fills readily — a little effort looks healthy.
Why gamma, not a capThis is the cleverest, most "product" decision in the app. A lower-stakes area looks healthy with less effort, so your eye is drawn to the areas you said matter most — a core area that's half-tended reads as noticeably less full than an optional one that's half-tended. Yet γ only bends the curve; it never caps below 1, so everything can still reach a full petal if you do it all. Gentle, but honest about priorities.

The full function, in order

function computeAreaActivity(area, habits, logs, windowDays = 14, endISO = today) {
  tendHabits = area's habits where polarity !== 'ease'
  if (tendHabits.length === 0) return 0            // "ready to grow" rest state
  from = endISO − (windowDays − 1) days

  // one pass: tally completed days per tend habit + total tug penalty
  for each log in [from, endISO]:
    if it's an ease habit: penalty += tugWeight ?? 1
    else: completedByHabit[log.habitId].add(log.date)

  totalWeight = Σ tend weights (or 1)
  positive = Σ (weight/totalWeight) × min(1, completed/expected)
  penaltyFrac = min(0.6, penalty × 0.12)
  score = max(0, positive − penaltyFrac)
  return min(1, score ^ FILL_GAMMA[area.importance])
}

Worked example

Area "Body" (importance = core, γ 1.35) with two equal-weight tend habits over 14 days:

  • "Walk" — daily → expected 14; logged 14 days → adherence 1.0
  • "Stretch" — 3× a week → expected 6; logged 3 days → adherence 0.5
  • One "some" tug logged once → penalty 1 → penaltyFrac 0.12

positive = ½·1.0 + ½·0.5 = 0.75 → score = 0.75 − 0.12 = 0.63 → fill = 0.631.35 ≈ 0.54.

The same numbers in a matters area (γ 1) would read 0.63; in an optional area (γ 0.6), 0.630.6 ≈ 0.75. Identical effort, three different fills — that's the attention dial at work.

Windows & reuse — one function, many surfaces

The same function drives multiple views by varying windowDays and endISO:

The endISO parameter was added specifically so the garden could re-bloom any historical week (ch. 08). The rolling window is why "no streaks" is structurally true: the calculation only ever looks at a trailing span, recalculated daily — a single missed day shifts a ratio slightly, it never "breaks" anything.

The caption & the geometry

Above the petals, a one-line caption reads the mean of all petal fills: high and even → "In a good rhythm today"; any petal below 0.25 → "A few areas waiting for you"; else "Tend to yourself today." Petals are laid out as wedges of a circle (custom SVG geometry), coloured by the area's own colour which is never themed — so the bloom always means the same thing regardless of the app's light.
