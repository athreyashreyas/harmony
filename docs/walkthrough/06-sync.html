Chapter 06 · the hard part

Sync Architecture

Offline-first sync that stays correct across devices without ever losing a write, and without re-downloading years of history. Four mechanisms: the durable outbox, the authoritative pull, realtime, and — for logs — watermark-based incremental sync with soft-deletes.

Principle: Dexie is the truth, the cloud is a mirror

Every read renders from Dexie. Every write goes to Dexie first (instant, optimistic) and is then propagated. The cloud never blocks the UI. This inverts the usual "server is truth" model and is the reason the app feels instantaneous and works fully offline.

1 · The durable outbox (never lose a write)

A write does two things atomically-ish: db.<table>.put(row) and outbox.add({op, table, payload}). A background flusher drains the outbox to Supabase in insertion order.

flushOutbox():
  for each queued item in id order:
    if op === 'delete' && table === 'logs':  // soft-delete (see §4)
      update logs set deleted_at = now() where id = ...
    else:
      upsert(table, payload)
    on server-rejection (has error.code): DROP the item (it's bad, don't block)
    on network error (no code):           STOP, retry later (keep the queue)
Why an outbox, not fire-and-forgetA tap on the subway must persist. The outbox makes writes durable and ordered: made offline, they queue and flush on reconnect; a failed request retries; and because ids increment, they drain in FK-safe order (an area before its habits before their logs). The distinction between "server rejected it" (drop) and "network failed" (retry) prevents one poison item from wedging the whole queue forever.

2 · The authoritative pull (server wins, gently)

On sign-in and as a backstop, the app fetches the account's rows and reconciles into Dexie with the server authoritative. The danger is deleting a local write that simply hasn't uploaded yet — solved by a grace window.

reconcileTable(local, serverRows):
  serverIds = set(serverRows.id)
  cutoff = now − RECONCILE_GRACE_MS (2 min)
  stale = local rows where NOT serverIds.has(id) AND writtenAt < cutoff
  bulkDelete(stale); bulkPut(serverRows)

Areas, habits, and settings always pull in full — they're tiny and their deletes must always propagate. The deletion decision is extracted to a pure, unit-tested staleIds().

3 · Realtime (instant cross-device)

Supabase Realtime broadcasts every row change for the user's rows; the app applies each straight into Dexie and reloads just the affected store. This is the instant path when both devices are online — a tap on your phone lands on your iPad in a moment. Backstops for anything realtime misses: a pull on foreground, on reconnect, and a slow 15-minute poll while visible.

4 · Watermark incremental sync for logs (the big one)

Logs are the one ever-growing table. Re-downloading them on every pull would blow the free-tier egress budget over years. The fix — and the app's most refined piece of infrastructure — is a watermark.

The schema (migration 0019)

  • logs.updated_at — a timestamp bumped on every write by a Postgres BEFORE INSERT OR UPDATE trigger.
  • logs.deleted_at — a soft-delete flag. A "delete" becomes an update setting deleted_at, so the removal is a change the sync can see rather than a silent disappearance.
  • An index on (user_id, updated_at) for the range scan.

The client algorithm

pullUserData(userId, full = false):
  watermark = full ? null : localStorage["harmony.logsSince.<userId>"]
  reseed = watermark == null || watermark < now − SAFE_RESEED_DAYS (45)

  if reseed:  fetch logs where deleted_at IS NULL        // live snapshot
              → snapshot-reconcile into Dexie
  else:       fetch logs where updated_at > watermark     // only the delta, tombstones included
              → partitionChanges(): bulkPut live, bulkDelete tombstoned

  watermark = latestTimestamp(watermark, fetched updated_at's)   // advance

So every reconnect/foreground pulls only what changed since last sync — inserts, edits, and deletions of any date — immediately and cheaply. A device that's never synced, or hasn't in 45+ days, reseeds the live snapshot once.

The pruner (bounded storage)

The worker hard-deletes tombstones older than 60 days, once an hour. Retention (60d) is deliberately longer than the client's reseed threshold (45d), so a device always reseeds before any tombstone it still needs could be pruned. Realtime also carries a live soft-delete as an UPDATE with deleted_at set, which the client treats as a removal.

Decision · why not a change-feed table or CRDTA separate sync_events table (double writes) or a full CRDT would be overkill for a single-user-with-a-few-devices app. Soft-delete + updated_at is the industry-standard "incremental sync with tombstones" and it's extensible: any future entity joins by adopting the same two columns and reusing the shared primitives (latestTimestamp, partitionChanges). It was explicitly built as the canonical sync channel so future edge cases club into it.

Guarantee: immediate + complete + cheap

ScenarioHow it stays correct
Both devices onlineRealtime — instant, any date, incl. deletes.

The seed distinction (a nuance to know)

On a fresh device (no local data), AuthGate calls pullUserData(userId, true) — a full seed so Dexie (and Insights, which reads all of it locally) has the whole history. A returning device uses the incremental path. Insights and the Bloom garden deliberately read the full log history from Dexie (a local, zero-egress read) rather than the 60-day UI window, so Year/All ranges and past-week blooms are complete.
