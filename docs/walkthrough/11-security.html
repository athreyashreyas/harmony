Chapter 11

Security & Privacy

Defence in depth: row-level security as the backbone, a zero-trust worker, strict secret isolation, and a set of hardening measures — plus a fully-designed (but deferred) zero-knowledge encryption plan.

Row-Level Security is the backbone

Every table has RLS enabled with a per-user policy — a user can only ever touch rows where auth.uid() = user_id, on select, insert, update, and delete (with check guards writes too). PostgREST also needs baseline GRANTs, which are explicit (migration 0007). This means even a fully-compromised client, using its own valid token, can only reach its own data — the isolation is enforced by Postgres, not by app code.

create policy "Users manage their own areas" on areas for all
  using (auth.uid() = user_id) with check (auth.uid() = user_id);

The zero-trust worker

The push worker holds the service-role key (which bypasses RLS), so it is written to never trust client input:

  • Every mutating endpoint (/subscribe, /delete-account) verifies the bearer token against Supabase and derives the user id from the token, never from the request body. You can only ever act on your own account.
  • The service-role key lives only in the Worker's env — it is never shipped to the browser. The browser only ever holds the public anon key.
  • Account deletion is admin-gated and lives in the worker: it deletes the user's data across all tables, then the auth.users row via the Admin API — a real deletion, not just data.

Auth flows

  • Password reset via a Supabase email link → set a new password on the reset screen.
  • Onboarding draft is kept in localStorage so a reload mid-signup never loses your areas; it's cleared on completion and on sign-out.
  • "What's new" is shown once per account: a synced lastSeenVersion (so seeing it on one device suppresses it elsewhere) and a per-device localStorage marker (which a mid-sync pull can never clobber), gated on a fresh pull so a stale value can't re-trigger it.

The deferred zero-knowledge encryption plan

A flagship privacy feature was fully designed and consciously deferred. The goal: the server stores only encrypted blobs, so Harmony can honestly say "we can't see your habits, reasons, or notes."

The architecture (envelope encryption)

  • A random Data Key encrypts content (AES-GCM, client-side Web Crypto — zero server cost).
  • The Data Key is stored wrapped by a key derived from the password (PBKDF2, ~600k iterations). The server never sees the password or the Data Key.
  • Changing the password re-wraps the Data Key (fast, no bulk re-encryption). A recovery key at sign-up wraps it a second time, so a forgotten password isn't automatic data loss.
The central trade-offThe worker reads your data to compose reminders and drift nudges (ch. 07). Full blackout would blind it and break those. The chosen path is "encrypt content, keep timings": encrypt names / "why" / notes, but keep cadence, reminder time, and log dates readable so reminders survive. The honest claim becomes "we can't read what you track or write, only the clock we nudge you by."
The browser-E2EE caveat (be honest in interviews)Because the server delivers the app's JavaScript, a malicious server could in theory serve key-leaking code. The defensible PWA claim is "stored only encrypted; we never transmit your keys or plaintext" — the absolute "even a hacked us couldn't" needs native apps + reproducible builds. Knowing this distinction is exactly the kind of nuance that impresses in a security conversation.

Cost verdict: E2EE is free-tier-safe for a decade — client-side crypto adds no server cost; storage/egress inflate only ~1.3–1.4× (well within headroom).
