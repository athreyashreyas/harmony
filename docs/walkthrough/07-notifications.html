Chapter 07

Notifications & the Push Worker

A Cloudflare Worker runs a pass every minute, sending three kinds of gentle nudge — habit reminders, an evening round-up, and drift nudges in your own words — using web push implemented from scratch with Web Crypto. This is the most "backend" part of the app.

Web push, from first principles

There's no library doing the crypto. The worker signs a VAPID JWT and encrypts each payload (aes128gcm) using the browser subscription's public key, entirely with the Workers runtime's SubtleCrypto — no Node dependencies. A subscription is { endpoint, p256dh, auth }, stored per device in push_subscriptions. When a push returns 404/410 (gone/expired), the worker deletes that subscription; unexpected statuses are logged, not silently dropped.

iOS realityWeb push on iOS requires the PWA be installed to the home screen, and iOS injects a "from Harmony" attribution line that the API can't remove. The app detects the install state and guides the user; the notification text is deliberately just Harmony + the message so it reads cleanly.

The every-minute pass

The cron handler runs runDriftPass. It prefilters to users with subscriptions, then processes them in parallel (Promise.allSettled) so one slow user can't stall the pass; each Supabase call and each push has a 10-second timeout for the same reason.

processUser(user):
  bundle = getUserBundle(user)        // areas, habits, last-60d live logs, 14d nudges, settings, subs
  if !bundle.settings.masterEnabled: return
  if bundle.subscriptions.length === 0: return
  inQuietHours = withinDnd(now, tz, dndStart, dndEnd)

  sendHabitReminders(...)             // user-chosen times — fire EVEN in quiet hours
  if inQuietHours: return             // the rest are ours to time → stay quiet
  sendDailySummary(...)
  sendDriftNudges(...)

1 · Habit reminders

For each habit with a reminderTime that is due today, still unlogged, and not already reminded today, at its set minute. "Due this minute" uses a catch-up window so a delayed cron run still delivers:

isDue(nowMin, targetMin, window) = (nowMin − targetMin) ≥ 0 AND < window
REMINDER_CATCHUP_MIN = 10 minutes

Paired with a once-per-habit-per-day guard (via NudgeHistory), a delayed or skipped run delivers a few minutes late rather than never — but never twice. Because they're times you chose, reminders fire even during quiet hours.

2 · The evening summary

One round-up of habits still unlogged that day, at 20:00 local (deliberately before the 21:00 quiet-hours default, so it isn't suppressed), with a 5-minute catch-up. Copy scales: "Walk is still waiting today. No rush." → "Still waiting today: A, B." → "5 things are still waiting today. Even one counts."

3 · Drift nudges (the soul of the feature)

When an area you said mattered goes quiet, Harmony brings your own sentence back to you. Capped at 2 drift nudges per user per day (this cap governs drift only; reminders and the summary have their own once-each guards).

The drift engine (detectDrift, shared)

Pure, and run both client-side (for the in-app banner) and server-side (for push) from the same code. It decides which areas are "quiet enough" to nudge, right now.

for each area (skip archived; skip importance === 'optional'):
  cadence   = areaCadence(area)     // median gap between log-days over 60d, clamped 1..14
                                     // (falls back to median habit cadence, then 7)
  daysSince = days since the area was last tended (or since it was created)
  base      = importance==='core' ? max(cadence×1.5, 5) : max(cadence×2.0, 10)
  threshold = base × sensitivity     // low 1.5, default 1, high 0.65
  if daysSince > threshold AND not firedRecently(area, 3 days):
    → candidate

Candidates are ranked core before matters, then by longest silence. Key honesty rule: tugs don't count as tending — a logged tug never resets an area's drift clock or shapes its cadence. Only tend-habit logs do.

Two independent dialsImportance sets the base threshold (core areas nag sooner: ×1.5 and a 5-day floor vs matters' ×2.0 and 10-day floor). Drift sensitivity is a separate per-area multiplier (high = 0.65× the threshold → sooner). Together they give fine control without a settings maze.

The template composer & library

Nudge text isn't hardcoded — it's composed from a library of templates (packages/shared/src/templates/), also shared and byte-identical between banner and push. A template is data with {placeholders}:

{ id: 'drift-words-days', type: 'drift',
  body: `You wrote: "{whySentence}" It's been {daysSince} days. Maybe {timeOfDay}.` }

Selection algorithm (compose):

  1. Filter templates by type, matching conditions (time of day, weekday, weather, min/max days-since, importance), and canUseTemplate — every placeholder it needs must have a value (weatherPhrase is exempt and simply drops out).
  2. Prefer templates not used recently for this area+type (the caller passes recent ids); if all used, allow a repeat rather than going silent.
  3. Weighted-random pick from the pool.

renderTemplate substitutes values, supports a {x.titleCase} modifier, and tidies the spacing an empty value leaves (e.g. a missing weather phrase). Placeholders resolve to your words (whySentence, areaName, firstName) or facts (daysSince, lastDayPhrase, timeOfDay). The app never speaks in its own voice.

Decision · server-composed content vs privacyBecause the worker composes messages from your data (habit names, "why" sentences, logs), it must be able to read that data. This is the central tension with the deferred zero-knowledge E2EE plan (ch. 11): true blackout would blind the server and break these nudges. The chosen future path is "encrypt content, keep timings" so reminders survive.

Reliability engineering

  • Catch-up windows (10m reminders / 5m summary) survive a delayed or skipped cron tick.
  • Parallel + timeouts: users, sends, and reminders run via Promise.allSettled; every Supabase/push call aborts after 10s.
  • Self-healing: the app re-subscribes a device whose permission is granted but whose backend row is missing; dead endpoints (404/410) are pruned.
  • Observability: a /health endpoint and structured logs of unexpected push statuses.
