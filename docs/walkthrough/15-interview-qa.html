Chapter 15 · the payoff

Interview Q&A Prep

The questions a sharp interviewer (or a curious user) will actually ask, with crisp answers you can say out loud. Practise these until they're reflex.

Product & design

How is this different from Streaks or Habitica?

Philosophy, made mechanical. There's no streak counter — the Bloom reads a rolling 14-day window recalculated daily, so a missed day shifts a ratio, it never "breaks." Negative habits (tugs) subtract from the picture so it's honest, not just wins. And nudges quote your own words back to you instead of the app lecturing. "No streaks, no shame" isn't a tagline — there is literally no field that stores days-in-a-row.

Why let bad habits ("tugs") lower the score? Isn't that discouraging?

A week showing only wins is a lie. Tugs make the Bloom honest — lift and drag together. But they're capped (a tug can never remove more than 60% of a petal) and floored at zero, and styled muted so they never read as an alarm. Honest, never punitive.

What's the single cleverest product idea in it?

The importance gamma on the Bloom. Lower-stakes areas fill more readily (γ 0.6) and core areas fill less readily (γ 1.35), so your eye is drawn to what you said matters most — yet gamma only bends the curve, so everything can still reach full. It encodes priority into the visual without any nagging.

Architecture

Walk me through what happens when I tap to log a habit.

Optimistically: write a Log to Dexie and update the Zustand store, so the check, count, and Bloom redraw before any network call. Enqueue an outbox item. A flusher drains it to Supabase; a DB trigger bumps updated_at. Realtime broadcasts it to my other devices, which apply it into their Dexie. If I was offline, the outbox retries on reconnect. The UI never waited on the network.

Why offline-first with the device as source of truth?

A tracker is opened constantly for two-second interactions and must work with no signal. Server-first would put the network in the critical path of every tap. Dexie-first makes it instant and fully offline-capable; the cloud reconciles in the background.

How do the client and the notification server stay consistent about "is this due"?

The scheduling and drift logic live in @harmony/shared, imported as source (no build) by both the web app and the Cloudflare worker. They run byte-identical code, so a reminder always matches the app's own view of the day.

The hard one — sync

How does cross-device sync work, and how do you keep it cheap over years?

Four mechanisms around Dexie: a durable ordered outbox for writes; an authoritative pull with a 2-minute grace window; Supabase Realtime for the instant path; and — for logs, the one growing table — watermark incremental sync. Each pull fetches only logs with updated_at > a per-device watermark. Deletes are soft-deletes, so a removal is a change the watermark can see rather than a silent gap. A device dormant 45+ days reseeds; the worker prunes tombstones after 60 days. Result: every reconnect catches up completely and immediately, transferring only the delta — bounded egress no matter how much history exists.

What about a conflict — two devices edit the same log?

Last-write-wins per row via upsert; realtime + the watermark converge them quickly. The grace window in the authoritative reconcile prevents the classic bug of deleting a local write that simply hasn't uploaded yet. For a single-user-with-a-few-devices app, LWW is the right simplicity; a CRDT would be over-engineering.

How do offline deletions propagate?

That's exactly why deletes are soft (a deleted_at flag), not hard. A hard delete leaves nothing for an offline device to catch up on. The soft-delete bumps updated_at, so the watermark pull applies the removal on the next sync — and realtime carries it live for online devices.

Security

How is one user's data isolated from another's?

Postgres row-level security: every table has a policy restricting all operations to rows where auth.uid() = user_id, with with check on writes. Even a compromised client with a valid token can only reach its own rows — enforced by the database, not app code.

The push worker has the service-role key — how do you stop it being abused?

Zero-trust: every mutating endpoint verifies the bearer token against Supabase and derives the user id from the token, never the body, so you can only act on your own account. The service-role key lives only in the worker's env, never in the browser. Plus a CORS allowlist, per-user rate limiting, and length CHECK constraints as defence in depth.

You want to say "we can't see your data" — can a web app truly promise that?

Honestly, with a caveat. The plan is envelope encryption: a random Data Key encrypts content, wrapped by a PBKDF2 key from your password; the server stores only ciphertext. But because the server ships the JS, a malicious server could serve key-leaking code — so the defensible PWA claim is "stored only encrypted; we never transmit your keys or plaintext," and the absolute version needs native apps + reproducible builds. Also, reminders need the server to read timings, so the pragmatic scope is "encrypt content, keep timings." Knowing those two nuances is the whole answer.

Scale & cost

Does this stay on the free tier as it grows?

For small scale, a decade, yes. Storage: ~1 MB/user/year of logs → ~100 MB for 10 heavy users over 10 years, under the 500 MB cap. Egress is bounded by the watermark sync. No server compute — the intelligence is client-side or in the shared engine the worker already runs. The one thing I'd add before hundreds of users is cron sharding, which I deliberately deferred because it would hurt reminder timeliness at current scale.

What breaks first at 10,000 users?

The every-minute worker pass scanning all active users — Cloudflare subrequest/CPU limits per invocation. The fix is sharding users across cron ticks (each user checked every N minutes), trading a little reminder precision for scale. Storage and the free Postgres tier would need upgrading well before the architecture does.

Reflective

What would you do differently?

I'd have started logs with updated_at + soft-delete from day one instead of arriving at it via a windowed pull then a daily self-heal — though those intermediate steps were reasonable given the requirements at each point. And I'd add tests around the impure Dexie reconcile, not just the pure core.

What are you most proud of?

Three things: the Bloom's gamma curve (priority encoded gently into a visual), the watermark sync (immediate, complete, and cheap all at once), and the reflection engine (it reads like a kind friend and never repeats, with zero randomness — purely a function of your state).
