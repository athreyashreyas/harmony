Chapter 02

Architecture & Stack

Harmony is an offline-first PWA with a tiny serverless backend. The defining architectural choice: the device is the source of truth, and the cloud is a mirror it reconciles against.

The monorepo

A pnpm workspace with three packages:

harmony/
  apps/
    web/          # the PWA (React 18 + TS + Vite)
    push-worker/  # Cloudflare Worker for web push (Web Crypto, no Node deps)
  packages/
    shared/       # types + scheduling + drift/template engine, shared by web AND worker
  supabase/
    migrations/   # SQL, applied by hand in the Supabase SQL editor
Decision · single-sourced logic@harmony/shared resolves to source (packages/shared/src/index.ts) via a tsconfig/Vite path alias — there is no build step for it. The web app and the push worker both import the exact same scheduling and drift code, so "is this habit due today?" and "has this area gone quiet?" are computed byte-identically on client and server. This is why a reminder fired by the worker always matches the app's own view.

The data-flow model (the important mental picture)

Read it as three arrows around Dexie:

1 · Write → Dexie → Outbox → Supabase. Every user action writes to Dexie first (instant, optimistic), and enqueues a durable outbox item. A flusher drains the outbox to Supabase in order. A write made offline (or one whose request fails) stays queued and retries — it is never lost.

2 · Pull (authoritative reconcile). On sign-in and as a backstop, the app fetches the account's rows and reconciles them into Dexie with the server authoritative, guarded by a grace window so a just-made local write is never deleted. (Ch. 06 details the watermark-based incremental version.)

3 · Realtime. Supabase Realtime pushes each row change to other devices instantly; the app applies it straight into Dexie and reloads the affected store.

Backstops for anything realtime misses (dropped socket, sleep): a pull on foreground, on reconnect, and a slow 15-minute poll while visible.

Why offline-first, not server-firstA habit tracker is opened dozens of times a day for a two-second tap. That interaction must be instant and must work on a subway with no signal. Making Dexie the truth means the UI never waits on the network — the network catches up in the background. It also makes the app resilient: Supabase could be down and you could still log your whole day.

Routing & app shell

React Router v6 with lazy-loaded screens. Public routes (/sign-in, /sign-up, /reset-password) sit outside the auth gate; everything else is wrapped in <AuthGate>, which orchestrates the boot (auth check, first pull, theme, and the "What's new" decision) behind a single calm splash. The main tabbed shell holds Home (index), Areas, Log, Insights, and Me.

// app/Router.tsx — the shape
/sign-in /sign-up /reset-password        // public
<AuthGate>
  /onboarding /guide /habit/:habitId
  <Shell>  // bottom-tab shell
    /            HomeScreen
    /areas       AreasScreen
    /log         LogScreen
    /insights    InsightsScreen
    /me          SettingsScreen

PWA specifics

  • Custom service worker (src/sw.ts, injectManifest) for push and notificationclick — the app shell is precached by Workbox, but the push logic is ours.
  • Seamless updates: a new version = a new SW bundle → controllerchange → reload, gated behind the "Updating Harmony" screen only when the version actually changed (ch. 12).
  • iOS reality: web push requires the PWA be installed to the home screen; the app detects this and guides the user. iOS also injects "from <app>" attribution on notifications — not removable via the API, a documented constraint.
