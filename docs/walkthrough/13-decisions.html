Chapter 13

Key Decisions & Trade-offs

The forks in the road, and why each was taken. This is the chapter to internalize — interviewers probe decisions, not features.

Decisions that were reversed (and why that's healthy)

  • 120-day windowed log pull → watermark sync. The window bounded egress but left an old-date-edit edge case with up to a day's delay. When "immediate sync, no matter what" became a hard requirement, it was replaced with the proper watermark + soft-delete mechanism. Lesson: the windowed pull was a good 80% step; the watermark is the correct 100%.
  • Daily self-heal → superseded. A once-a-day full pull was implemented to close the edge, then removed the same session because the watermark closes it immediately, not daily.
  • Tap-to-expand chevron → wrap-in-place. A cropped long habit name first got a tap-to-reveal popover, then a tap-to-expand chevron — both misfired on touch (opening the habit page). The final answer removed the interaction entirely: long names just wrap. Lesson: the best affordance was no affordance.
  • Ocean Blue removed. Trimmed to a curated 5-light / 3-dark set to avoid choice fatigue; stale selections fall back to the default gracefully.

Deliberate non-decisions (things intentionally NOT built)

  • Cron sharding across users. It would make reminders fire less often per user, hurting the "within a minute" guarantee. Deferred until scale actually demands it — a case where not doing the "scalable" thing was the right call.
  • Full zero-knowledge blackout. Would break server reminders; the pragmatic middle path preserves functionality (ch. 11).
  • A charting/animation-heavy dependency. Kept out to protect bundle size and the free-tier egress budget.
The meta-lessonNearly every decision optimizes for one of three things: the calm philosophy (no shame, your words), instant offline UX (Dexie-first), or staying free at scale (bounded egress, no server compute, no deps). When a new idea conflicts with all three, it's reshaped until it doesn't — that's the design method.
