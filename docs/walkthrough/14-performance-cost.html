Chapter 14

Performance & Free-Tier Cost

Harmony is built to run free — Supabase and Cloudflare free tiers — for a decade at small scale. The two levers are egress (data transferred) and storage; the design bounds both.

Client performance (feels instant)

  • Optimistic everything. Every write hits Dexie + the store before any network call — the UI never awaits the round-trip.
  • Lazy routes. Each screen is a code-split chunk; the initial bundle is small.
  • Bounded reads. The Home/day stores hold a trailing 60-day window of logs; Insights loads the full history but only from local Dexie (no egress).
  • Memoized compute. The Bloom, insights, reflection, and garden are all useMemo'd on their inputs; the garden pre-groups logs by area so its weeks-×-areas pass stays near-linear.
  • Indexed queries. Dexie compound index [habitId+date] makes "logged today?" O(1)-ish; server has a (user_id, updated_at) index for the watermark scan.

Egress — the real cost driver, and how it's bounded

The threat was: re-downloading the whole log history on every sync. The fixes:

BeforeAfter
select('*') on all logs on every pull (sign-in + focus + reconnect + poll)Watermark incremental pull — only logs changed since last sync (ch. 06)
5-minute poll, each a full pull15-minute poll, each a tiny delta (realtime carries the live path)
Insights implicitly wanted all logs over the wireInsights reads full history from local Dexie — zero egress

Net: recurring egress is now bounded regardless of how many years of logs accumulate. A returning device downloads only its delta; a fresh device seeds once.

Storage — the decade math

A Postgres log row is ~250 bytes with indexes. A heavy user logging ~10/day ≈ 3,650 rows/year ≈ ~1 MB/user/year.

10 heavy users × 10 years ≈ ~100 MB, against Supabase's 500 MB free cap. Comfortable for a decade.

Growth watch-items: nudge_history (only ~60 days needed for anti-spam — prune) and log tombstones (soft-deletes) — the worker prunes those older than 60 days, hourly, so they never pile up.

No server compute cost

  • All the "intelligence" (Bloom maths, drift detection, insights, reflection) runs client-side or in the shared engine the worker already runs — no extra services, no per-request compute bill.
  • The push worker is a cron + a few PostgREST calls; parallelized with timeouts so a slow user can't run up wall-time.
  • The deferred E2EE would keep this property: encryption is browser Web Crypto, adding zero server cost (storage/egress inflate only ~1.3–1.4×, still within headroom).

Correctness = performance (a subtle point)

The audit that produced these fixes also caught a correctness bug hiding in a perf boundary: because Insights read the 60-day store, Year/All ranges and the Bloom garden silently under-reported beyond 60 days. Switching Insights to read full local history fixed the feature and kept egress at zero — a case where the performance-minded fix (read locally) was also the correctness fix.

Testing as a safety net

The riskiest logic — scheduling, the Bloom, drift, templates, insights analytics, the reflection, and the sync reconcile core — is covered by a Vitest suite (100+ tests) of pure functions. The impure edges (Dexie, network) are kept thin around a tested pure core (e.g. staleIds, latestTimestamp, partitionChanges for sync). This is why sync could be reworked twice in a session without losing confidence.
