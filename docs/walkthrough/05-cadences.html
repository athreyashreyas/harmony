Chapter 05

Scheduling & Cadences

Two pure functions in packages/shared/src/scheduling.ts answer every scheduling question in the app: isHabitDueOn ("is this on today's list?") and expectedCompletionsInWindow ("how many should there be?"). Because they're shared, the app's "today" and the worker's reminders always agree.

isHabitDueOn(habit, dateISO)

Guard clauses first, then the cadence switch:

if (habit.archivedAt != null) return false
if (habit.polarity === 'ease') return false      // tugs are NEVER scheduled
if (dateISO < habit.startDate) return false
if (habit.endDate != null && dateISO > habit.endDate) return false

switch (cadence.kind) {
  daily:          return true
  weekdays:       return dayOfWeek in 1..5
  weekends:       return dayOfWeek === 0 || 6
  specific-days:  return cadence.days.includes(dayOfWeek)
  times-per-week: return true                     // shown every day; you pick which
  every-n-days:   return daysBetween(start, date) % max(1,n) === 0
  every-n-weeks:  return daysBetween(start, date) % (7 × max(1,n)) === 0
  every-n-months: // due on the start day-of-month, clamped to month length
}
The month-clamp subtletyFor every-n-months, a habit that starts on the 31st is "due" on the 28th in February (or 30th in April) — it clamps to the month's last day so a monthly habit never silently skips a short month. This is the kind of edge that's unit-tested explicitly.

expectedCompletionsInWindow(cadence, windowDays)

How many completions a cadence implies over a span. This is the denominator for every adherence ratio (the Bloom, insights, the garden). It's an approximation from the cadence shape, not a day-by-day walk — because the windows in practice are whole weeks.

Where "today's list" comes from

Home's list is habits.filter(isHabitDueToday) where isHabitDueToday is just isHabitDueOn(habit, todayISO()). The Today/All toggle switches between due-today and all-kept. Tugs are excluded from this list entirely (they're never scheduled) and live in their own section.

Dates are days, not moments

A subtle but important convention: dates that represent a day (a log's date, start/end dates) are ISO strings "YYYY-MM-DD" in local time, produced by todayISO(). This avoids the classic UTC bug where logging at 11pm lands on "tomorrow." Helpers daysBetween, isoDaysAgo, and daySegment (morning/afternoon/evening/night by local hour) all live in shared so client and worker compute identically.

Why this is pure & sharedScheduling is the highest-leverage correctness code in the app — if the worker thought a habit was due when the app didn't, you'd get a reminder for something not on your list. Keeping it pure (no I/O, deterministic given inputs) makes it trivially unit-testable, and keeping it in @harmony/shared guarantees the client and the Cloudflare worker run the exact same bytes.
